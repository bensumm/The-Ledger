## Overall finding

Most of the proposed dispositions are **correct** (9 of 14 confirmed outright). Three are
**wrong or materially misframed**:

1. **#4 `alertCount` — proposed DELETE is WRONG; the right call is WIRE (A).**
   `js/watch.js:285` duplicates its exact expression inline
   (`heldVerdicts.filter(isHeldAlert).length + offerVerdicts.filter(v => v === CANCEL_BID).length`),
   while the `watchcore.js` header above `alertCount` declares "The tab badge AND the summary
   'Alerts' cell both read this ONE count — never diverge them." The flagged export is the
   intended one-home; the production path is the fork. Deleting it deletes a fixture-pinned
   contract while leaving the duplication in place.
2. **#9 `STATUS` — proposed TAG is WRONG; delete it (C).** It has been dead since the commit
   that created it (`910bea1`, V2-P2). The `'pass'|'caution'|'reject'` string contract is the
   documented API (validate.mjs header line 7) and is used as bare literals ~30× inside
   validate.mjs itself and by every consumer (`screen.mjs:454/459/902/904`). Tagging would
   enshrine a vocabulary constant that even its own module never adopted.
3. **#11 `modules.mjs` framework — the KEY QUESTION's premise (and the shipped `@test-only`
   marker text) is FALSE.** The probe-orchestration framework **IS adopted in production**:
   `screen.mjs` (:108 import, :647 `runProbes`, :655 `logFirings`, :1231 `loadModules`) and
   `quote.mjs` (:35, :116, :258, :265) both drive it, four real probes exist
   (`pipeline/modules/{dip,froth,anchor,decant}.mjs`), and the firing logs hold thousands of
   real production firings (dip.log ~3.5k lines, anchor.log ~12k). The marker on
   `collectNeeds` (modules.mjs:173) claims "the probe-orchestration framework is not yet wired
   into a production surface" — that reason is factually wrong and should be corrected.
   Only `collectNeeds`' *active pre-fetch* contract is unconsumed (deliberate, documented).

One premise correction: **#10 `STAGES` is not actually dead** — it is internally load-bearing
(stage validation, `modules.mjs:111`, two occurrences), so the guard's own heuristic (b)
classes it LIVE and it carries no marker today. Nothing to do.

Two smaller notes surfaced in passing: (a) `reconcileDipPool` **lost** `selectNominations`'
legacy *name*-dedup — a manual legacy string entry ("Blood rune") and a new auto qualifier for
the same item can now coexist as duplicates in the pool (minor; the fix, if wanted, belongs in
`reconcileDipPool`, not in keeping the legacy fn). (b) The N1 delete cluster reaches into
**app-imported** `js/estimators.mjs` (the `'rising'` estimator family), so the audit's
"node-only ⇒ no APP_VERSION" line is slightly too broad — the deletion is still behavior-neutral
for the app (no spec names the family), so the `flushSignal`/`nominateDip` no-bump precedent
applies, but the executor should know they are editing an app-served file.
